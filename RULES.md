# RULES.md

# Правила разработки проекта

Версия: 1.0

---

# ОБЯЗАТЕЛЬНО ПРОЧИТАТЬ

Перед выполнением любой задачи необходимо:

1. Полностью прочитать PROJECT.md.
2. Проверить актуальный статус TASKS.md.
3. Только после этого приступать к работе.

---

# ГЛАВНОЕ ПРАВИЛО

Не принимать архитектурных решений самостоятельно.

Если существует несколько вариантов реализации:

Сначала предложить варианты.

Не менять архитектуру без явного согласия пользователя.

---

# ПРИНЦИПЫ РАЗРАБОТКИ

Проект должен быть:

• понятным

• расширяемым

• модульным

• легко поддерживаемым

• современным

• быстрым

Код пишется так, будто проект будет развиваться много лет.

---

# МОДУЛЬНАЯ АРХИТЕКТУРА

Каждая функция должна находиться в собственном модуле.

Не создавать огромные файлы.

Не смешивать разные функции.

Например

Практики

↓

отдельный модуль

Таймер

↓

отдельный модуль

AI

↓

отдельный модуль

Статистика

↓

отдельный модуль

Дневник

↓

отдельный модуль

Материалы

↓

отдельный модуль

Настройки

↓

отдельный модуль

---

# НЕ ЛОМАТЬ РАБОЧИЙ КОД

Если функция уже работает —

не переписывать её полностью.

Вносить только необходимые изменения.

Минимизировать риск появления новых ошибок.

---

# НЕ ПЕРЕПИСЫВАТЬ ПРОЕКТ

Не менять:

структуру проекта

архитектуру

организацию модулей

без согласования.

---

# НЕ СОЗДАВАТЬ ДУБЛИРОВАНИЕ

Если функциональность уже существует —

использовать её повторно.

Не копировать одинаковый код.

---

# ДОКУМЕНТИРОВАНИЕ

Каждый сложный участок кода должен иметь комментарий.

Каждый новый модуль должен иметь краткое описание своего назначения.

---

# ПРОСТОТА

Предпочитать простые решения.

Не использовать сложные технологии,

если простое решение полностью решает задачу.

---

# ПРОИЗВОДИТЕЛЬНОСТЬ

Не оптимизировать преждевременно.

Сначала сделать рабочую функцию.

Потом оптимизировать при необходимости.

---

# AI

ИИ используется только как помощник.

ИИ не должен управлять:

логикой приложения

хранением данных

таймерами

уведомлениями

статистикой

Если AI недоступен,

приложение должно продолжать работать.

---

# OPEN SOURCE

Архитектура должна позволять заменить AI-модель без изменения остального проекта.

Использовать единый AI Adapter.

Не привязывать проект к одному поставщику.

---

# МАТЕРИАЛЫ

Использовать материалы только из следующих источников:

https://www.advayta.org/

https://institute-vasishtha.com/

https://sanatanadharma.world/ru

Не копировать статьи.

Не хранить их внутри проекта.

Хранить только ссылки.

---

# UI / UX

Главный критерий —

приложением приятно пользоваться каждый день.

Интерфейс должен быть:

современным

понятным

красивым

быстрым

адаптированным под мобильные устройства.

---

# ДИЗАЙН

Использовать:

спокойные цвета

качественные иконки

карточки

современную типографику

мягкие анимации

тёмную и светлую темы

Тематическое оформление:

природа

горы

небо

мандалы

медитативная атмосфера

---

# ПЕРЕД ДОБАВЛЕНИЕМ НОВОЙ ФУНКЦИИ

Ответить на вопросы:

1.

Есть ли уже подходящий модуль?

Если да —

расширить его.

2.

Если нет —

создать новый модуль.

3.

Не нарушит ли изменение существующую архитектуру?

---

# ПЕРЕД ЗАВЕРШЕНИЕМ ЛЮБОЙ ЗАДАЧИ

Убедиться, что:

✔ код компилируется

✔ нет ошибок

✔ нет предупреждений

✔ не сломан существующий функционал

✔ обновлён TASKS.md

✔ код соответствует PROJECT.md

---

# ПОСЛЕ КАЖДОЙ ЗАВЕРШЁННОЙ ЗАДАЧИ

Необходимо:

1.

Обновить TASKS.md

2.

Кратко описать,

что было сделано.

3.

Предложить следующую логичную задачу.

---

# ЗАПРЕЩЕНО

Не удалять рабочий код без необходимости.

Не менять архитектуру.

Не использовать случайные библиотеки без объяснения.

Не добавлять функции, которых нет в PROJECT.md, без согласования.

Не усложнять проект без причины.

---

# КАЧЕСТВО КОДА

Предпочитать:

простые функции

понятные названия

небольшие файлы

хорошую читаемость

низкую связанность модулей

высокую расширяемость.

---

# ФИЛОСОФИЯ ПРОЕКТА

Это не просто приложение.

Это личный помощник для ежедневной духовной практики.

Любое решение должно отвечать на вопрос:

"Сделает ли это ежедневную практику проще, приятнее и полезнее?"

Если ответ "нет" —

такую функцию добавлять не нужно.

---

# ГЛАВНОЕ ПРАВИЛО

Качество важнее количества.

Лучше реализовать одну функцию идеально,

чем десять функций посредственно.

Проект должен развиваться постепенно,

сохраняя чистую архитектуру и высокое качество кода.
