# DEVELOPMENT ROADMAP

Дата составления: 2026-07-11

Ниже — пошаговый план доработки проекта, разбитый на небольшие и проверяемые этапы. Каждый этап содержит цель, изменяемые файлы (ориентировочно) и ожидаемый результат.

=== Этап 0 — Подготовка (быстрые выиграши)
- Цель: привести окружение в порядок, добавить базовые инструкции и переменные окружения.
- Файлы: `package.json` (root), `apps/*/package.json`, `.env.example`, `README.md` (новые).
- Изменения: добавить `.env.example`, короткий README с командами `npm install`, `npm run dev:api`, `npm run dev:mini-app`; добавить `start`/`dev` скрипты при необходимости.
- Результат: разработчик может запустить API, bot и mini-app локально по инструкции.

=== Этап 1 — Безопасность и auth (высокий приоритет)
- Цель: обеспечить надёжную идентификацию пользователя из Mini App и безопасный доступ к API.
- Файлы: `apps/api/src/http.ts`, `apps/api/src/modules/auth` (новый модуль), `apps/api/src/container.ts`, `apps/bot/src/config.ts`.
- Изменения:
  - Добавить модуль `auth` для валидации Telegram init data / подписи.
  - Источник userId в `ApiRequest.context` получать через авторизацию, а не доверять `x-user-id` по умолчанию.
  - Внедрить middleware / проверку прав в критичных сервисах (schedule, practices).
- Результат: Mini App может безопасно аутентифицировать пользователя; операции ограничены owner userId.

=== Этап 2 — Валидация и стабильность API
- Цель: добавить строгую валидацию запросов и ясную обработку ошибок.
- Файлы: `apps/api/src/routes.ts`, добавить `libs/validation` или `packages/shared` (зависимости), `apps/api/src/types.ts`.
- Изменения:
  - Подключить `zod` или `io-ts` и валидировать body/query для всех маршрутов.
  - Возвращать 4xx/5xx коды корректно; добавить centralized error handler.
- Результат: API устойчив к неверным запросам, понятные ошибки для клиента.

=== Этап 3 — AI: интеграция и graceful-fallback
- Цель: подключить реального AI-провайдера через конфиг и оставить mock как fallback.
- Файлы: `packages/ai-adapter/*`, `apps/api/src/container.ts`, `apps/api/src/modules/schedule/ai-service.ts`.
- Изменения:
  - Добавить конфиг `AI_PROVIDER=mock|openai` в env и использовать соответствующий провайдер в контейнере.
  - В `ScheduleAiService` и других местах вернуть контролируемые ошибки/результаты при неудаче парсинга.
- Результат: функционал создания расписания текстом/голосом работает при корректной настройке API-ключа; без ключа приложение работает с fallback.

=== Этап 4 — UI/UX и детали Mini App
- Цель: доработать интерфейс, формы, workflow завершения практики и настройки.
- Файлы: `apps/mini-app/src/*` (App.tsx, components screens, styles.css).
- Изменения:
  - Доработать Settings экран (уведомления, тема, AI toggle).
  - Обработать edge-cases у таймера (конфликты с backend) и улучшить UX для создания расписания из текста.
  - Добавить формы с фронтенд-валидацией и полезные подсказки.
- Результат: более гладкий пользовательский путь; меньше ошибок от некорректных данных.

=== Этап 5 — Уведомления и scheduler
- Цель: обеспечить надёжную отправку и управление уведомлениями (retry, backoff, idempotency).
- Файлы: `apps/bot/src/*`, `apps/api/src/modules/notifications/*`, `packages/database` миграции.
- Изменения:
  - Добавить retry/mark-as-sent/cancel логику и фоновый планировщик (simple cron/poll механизм уже в `bot`).
  - Убедиться, что jobs idempotent и репозитории корректно помечают `sent`/`cancelled`.
- Результат: надёжные пуши и контроль состояния уведомлений.

=== Этап 6 — Тесты, CI и качество
- Цель: покрыть критичные компоненты тестами и настроить CI.
- Файлы: `.github/workflows/*`, `packages/*/test`.
- Изменения:
  - Добавить unit тесты для `packages/domain` и интеграционные тесты API с in-memory/temporary sqlite.
  - Настроить GH Actions: lint, typecheck, test, build.
- Результат: автоматическая проверка PR и снижение риска регрессий.

=== Этап 7 — Документы и релиз
- Цель: подготовить релизный набор и документацию для деплоя.
- Файлы: `README.md`, `DEPLOY.md`, `Dockerfile`, `docker-compose.yml`.
- Изменения: инструкции по развертыванию, prod-конфиг, sample env, backup DB.
- Результат: готовность к размещению на VPS/cloud.

---

Каждый этап разбивать на задачи (issues/PR), проверять на dev окружении, делать code review и оставлять краткий отчёт в `TASKS.md` после выполнения (следуя RULES.md).
